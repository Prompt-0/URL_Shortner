## 2024-05-24 - [Information Disclosure in Protected Links]
**Vulnerability:** The API endpoint `api::get_link` and the UI endpoint `handlers::stats` exposed the `original_url` (the target destination) of a shortened link even if that link was protected by a password.
**Learning:** Partial implementation of security controls can lead to unauthorized data disclosure. While the redirect handler prevented the redirect (albeit with a 400 instead of 401), the other handlers returning information about the link did not verify the authorization status (password presence). We need to verify authorization logic uniformly across all endpoints returning object details.
**Prevention:** Always verify authorization checks across all points of access to a resource. When introducing a security control like password protection, ensure it covers not only the primary action (redirecting) but also secondary actions like viewing metadata or statistics. Fail securely with correct HTTP statuses (e.g., 401 Unauthorized).
